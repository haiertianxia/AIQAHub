# AIQAHub 产品架构设计

## 1. 产品定位

**AIQAHub** 是一个 AI 驱动的智能质量管控平台，旨在为软件工程团队提供统一的文档质量审查、测试结果聚合、覆盖率分析和质量门禁能力。

> **定位调整说明**：测试用例管理、测试计划、测试执行等功能已在 devops-platform 中包含，AIQAHub 聚焦于质量管控核心能力。

### 1.1 核心价值主张

- **统一质量管控平面**：将分散的文档审查、测试结果、覆盖率数据聚合到单一平台
- **文档质量审查**：产品文档、技术文档、API 文档的自动化 + 人工评审
- **AI 增强分析**：利用 AI 自动分析文档问题、测试失败根因、提供修复建议
- **质量门禁管控**：通过多维度质量规则管控整个研发流程（PR、发布等）
- **多维度治理可见性**：完整的审计追踪、事件流监控和质量指标看板

### 1.2 目标用户

| 用户角色 | 核心需求 | 使用场景 |
|---------|---------|---------|
| **产品经理** | PRD 质量审查、需求完整性检查 | 产品文档提交前评审、版本对比 |
| **架构师/技术负责人** | 技术方案评审、架构设计审查 | 技术文档评审、设计合规性检查 |
| **开发工程师** | API 文档审查、质量门禁反馈 | 提交代码后门禁检查、文档自检 |
| **QA 负责人** | 质量趋势分析、覆盖率缺口分析 | 周会质量汇报、风险识别、规则配置 |
| **DevOps 工程师** | 质量门禁集成、通知策略配置 | CI/CD 集成、环境维护、告警配置 |

## 3. 核心功能模块详解

### 3.1 文档审查模块 (Document Reviews)

**核心能力：**
- 文档上传与版本管理
- AI 预评审（自动检查完整性、规范性）
- 人工评审工作流
- 评审意见与评论
- 评审状态流转
- 文档评分与质量报告

**文档类型：**
- **产品文档**：PRD、需求文档、用户故事
- **技术文档**：架构设计、技术方案、接口设计
- **API 文档**：OpenAPI/Swagger、GraphQL Schema

**关键实体：**
- Document: 文档主体
- DocumentVersion: 文档版本
- ReviewTask: 评审任务
- ReviewComment: 评审评论
- ReviewChecklist: 评审检查项
- ReviewScore: 评审评分

**评审工作流：**
```
DRAFT → SUBMITTED → AI_REVIEWING → AI_REVIEWED → HUMAN_REVIEWING → REVIEWED → APPROVED / REJECTED
                                      ↓                                      ↓
                                 AI_FAILED                            NEEDS_REVISION
```

### 3.2 覆盖率分析模块 (Coverage Analysis)

**核心能力：**
- 覆盖率数据采集（Jacoco、Coverage.py、Istanbul 等）
- 覆盖率快照存储
- 覆盖率趋势分析
- 覆盖率缺口识别
- 覆盖率对比（分支/提交/版本间）
- 覆盖率质量门禁规则

**覆盖率维度：**
- 行覆盖率 (Line Coverage)
- 分支覆盖率 (Branch Coverage)
- 函数覆盖率 (Function Coverage)
- 语句覆盖率 (Statement Coverage)

**关键实体：**
- CoverageSnapshot: 覆盖率快照
- CoverageMetric: 覆盖率指标
- CoverageTrend: 覆盖率趋势
- CoverageGap: 覆盖率缺口

### 3.3 质量门禁模块 (Quality Gates)

**核心能力：**
- 多维度门禁规则定义
- 规则版本管理与审计
- 门禁评估与结果判定
- 门禁事件触发通知
- 与研发流程集成（PR、发布）

**规则类型：**

| 类别 | 规则示例 | 说明 |
|-----|---------|------|
| **文档质量** | `doc_review_status == APPROVED` | 文档必须评审通过 |
| **文档评分** | `doc_score >= 80` | 文档评分不低于 80 分 |
| **测试通过率** | `test_pass_rate >= 95%` | 测试通过率不低于 95% |
| **测试失败数** | `test_failure_count <= 5` | 失败用例数不超过 5 个 |
| **覆盖率** | `coverage_line >= 80%` | 行覆盖率不低于 80% |
| **覆盖率变化** | `coverage_line_delta >= -2%` | 覆盖率下降不超过 2% |
| **AI 风险** | `ai_risk_score <= 3` | AI 风险评分不高于 3 |

**关键实体：**
- QualityRule: 质量规则
- QualityRuleRevision: 规则修订历史
- GateEvaluation: 门禁评估记录
- GateCheck: 门禁检查项

**门禁应用场景：**
- **PR 门禁**：代码合并前必须通过门禁
- **发布门禁**：版本发布前必须通过门禁
- **定时门禁**：定期评估项目质量健康度

### 3.4 测试结果聚合 (Test Results)

**核心能力：**
- 多格式报告解析（JUnit XML、Playwright JSON、Allure 等）
- 报告数据归一化与聚合
- 测试结果趋势分析
- 失败分类与统计
- 多次执行对比

> **说明**：测试执行在 devops-platform 中完成，AIQAHub 专注于结果聚合与分析。

**关键实体：**
- TestReport: 测试报告
- TestCaseResult: 测试用例结果
- TestSuiteResult: 测试套件结果

### 3.5 AI 分析模块 (AI)

**核心能力：**
- **文档智能分析**
  - 文档完整性检查
  - 文档规范性建议
  - 文档质量评分
  - 需求歧义识别

- **测试失败分析**
  - 失败根因自动分析
  - 修复建议生成
  - 类似失败案例关联

- **质量风险预测**
  - 代码变更风险评估
  - 发布风险预警

**关键实体：**
- AIInsight: AI 洞察记录
- DocumentAnalysis: 文档分析结果
- FailureAnalysis: 失败分析结果

### 3.6 项目管理 (Projects)

**核心能力：**
- 项目元数据管理（名称、描述、标签）
- 项目级配置隔离
- 项目级通知策略覆盖

**关键实体：**
- Project: 项目主体

### 3.7 通知中心 (Notifications)

**核心能力：**
- 多通道支持（邮件、钉钉、企业微信、Slack）
- 策略化路由（按事件类型、项目、级别）
- 发送失败回退机制
- 通知模板管理
- 发送历史与审计

**关键配置：**
- `notification_default_channel`: 默认通道
- `notification_policies`: 策略数组（JSON）
  - `scope_type`: global / project
  - `scope_id`: 作用域 ID
  - `event_type`: 事件类型
  - `channels`: 目标通道
  - `filters`: 过滤条件

**事件类型：**
- `doc.review_required` - 文档待评审
- `doc.reviewed` - 文档评审完成
- `gate.failed` - 门禁失败
- `gate.passed` - 门禁通过
- `ai.analysis_ready` - AI 分析完成
- `coverage.warning` - 覆盖率告警

### 3.8 治理中心 (Governance)

**核心能力：**
- 平台运营概览仪表盘
- 事件流实时监控
- 通知投递追踪
- 审计日志查询
- 配置变更历史
- 连接器健康状态

**监控分区：**
- Overview: 关键指标总览
- Document Reviews: 文档评审概况
- Coverage Trends: 覆盖率趋势
- Quality Gates: 门禁执行情况
- Event Stream: 事件时序流
- Notification Events: 通知投递详情
- Audit Log: 操作审计

## 5. 产品成功指标

### 5.1 业务指标
- **文档评审覆盖率**: 接入评审的文档数 / 总文档数
- **平台采纳率**: 活跃项目数 / 总项目数
- **门禁执行率**: 接入门禁的流程数 / 总流程数
- **用户活跃度**: 日活/周活/月活用户数
- **通知到达率**: 成功发送通知数 / 总通知数

### 5.2 质量指标
- **文档平均评分**: 所有评审文档的平均分趋势
- **门禁拦截率**: 成功拦截的坏提交/发布数 / 总坏提交/发布数
- **AI 分析准确率**: 用户采纳的 AI 建议数 / 总 AI 建议数
- **平均修复时间 (MTTR)**: 从问题发现到修复的平均耗时
- **覆盖率达标率**: 覆盖率达标的项目数 / 总项目数

### 5.3 效能指标
- **文档评审周期**: 从提交到完成的平均耗时
- **门禁评估耗时**: 从触发到结果返回的耗时
- **报告生成延迟**: 执行完成到报告可用的耗时

## 6. 与 devops-platform 的关系

| 功能领域 | devops-platform | AIQAHub | 集成方式 |
|---------|----------------|---------|---------|
| **测试用例管理** | ✅ 核心功能 | ❌ 不涉及 | - |
| **测试计划** | ✅ 核心功能 | ❌ 不涉及 | - |
| **测试执行** | ✅ 核心功能 | ❌ 不涉及 | - |
| **测试结果聚合** | ⚠️ 基础展示 | ✅ 深度分析 | Webhook / API |
| **文档管理** | ❌ 不涉及 | ✅ 核心功能 | - |
| **文档评审** | ❌ 不涉及 | ✅ 核心功能 | - |
| **覆盖率分析** | ❌ 不涉及 | ✅ 核心功能 | 数据推送 |
| **质量门禁** | ⚠️ 简单规则 | ✅ 多维度规则 | PR / 发布集成 |
| **AI 分析** | ❌ 不涉及 | ✅ 核心能力 | - |

**协作流程示例**：
1. devops-platform 执行测试 → 2. 结果推送到 AIQAHub → 3. AIQAHub 进行深度分析 + 门禁评估 → 4. 结果回调 devops-platform → 5. devops-platform 执行流程阻断/放行
